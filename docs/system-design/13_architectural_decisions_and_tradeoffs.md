# Architectural Decisions and Tradeoffs

## 核心架构决策

### 决策 1: 单进程架构

**[FACT]** 选择：单 Python 进程 + asyncio

**收益**:
- 简单部署（无需编排）
- 无分布式复杂性
- 状态管理简单
- 开发调试容易

**代价**:
- 无水平扩展
- 单点故障
- 并发受限（全局锁）
- 资源受限于单机

**替代方案**:
- 多进程 + 消息队列（如 Celery）
- 微服务架构
- Actor 模型（如 Ray）

**为何选择当前方案**:
- 符合"ultra-lightweight"目标
- 个人/小团队使用足够
- 避免过度工程

---

### 决策 2: 文件系统存储

**[FACT]** 选择：JSONL + Markdown 文件

**收益**:
- 零依赖（无需数据库）
- 人类可读
- 易于备份/迁移
- grep 友好

**代价**:
- 无事务保证
- 并发写入风险
- 查询性能差
- 无索引

**替代方案**:
- SQLite（轻量级数据库）
- PostgreSQL（完整数据库）
- Redis（内存存储）

**为何选择当前方案**:
- 极简主义
- 适合低频访问
- 透明性（可直接查看文件）

---

### 决策 3: LiteLLM 抽象层

**[FACT]** 选择：LiteLLM 作为主要 provider

**收益**:
- 统一 API（20+ providers）
- 自动重试和错误处理
- 社区维护
- 快速支持新 provider

**代价**:
- 额外依赖
- 抽象层开销
- 某些 provider 特性不支持
- 调试困难（多层封装）

**替代方案**:
- 直接调用各 provider SDK
- 自建抽象层
- 只支持少数 provider

**为何选择当前方案**:
- 快速支持多 provider
- 减少维护负担
- 社区生态

---

### 决策 4: 全局处理锁

**[FACT]** 选择：单个 asyncio.Lock 保护消息处理

**收益**:
- 避免竞态条件
- 实现简单
- Session 状态一致性

**代价**:
- 串行处理（低吞吐）
- 一个慢请求阻塞所有
- 无法利用多核

**替代方案**:
- 细粒度锁（per-session）
- 无锁设计（actor 模型）
- 多进程

**为何选择当前方案**:
- 正确性优先
- 简单实现
- 个人使用场景吞吐量足够

---

### 决策 5: Memory 整合机制

**[FACT]** 选择：LLM 驱动的自动总结

**收益**:
- 智能提取关键信息
- 自然语言输出
- 灵活适应不同内容

**代价**:
- 额外 LLM 调用成本
- 可能丢失细节
- 不确定性（LLM 输出）

**替代方案**:
- 简单截断（丢弃旧消息）
- 规则提取（关键词）
- 向量数据库（语义搜索）

**为何选择当前方案**:
- 保留语义信息
- 适合对话场景
- 利用 LLM 能力

---

### 决策 6: Channel 适配器模式

**[FACT]** 选择：每个平台独立实现 BaseChannel

**收益**:
- 平台特性完整支持
- 解耦（修改一个不影响其他）
- 易于添加新平台

**代价**:
- 代码重复（相似逻辑）
- 维护成本高（11 个实现）
- 测试复杂

**替代方案**:
- 统一 webhook 接口
- 第三方聚合服务（如 Botpress）
- 只支持少数平台

**为何选择当前方案**:
- 最大灵活性
- 平台特性不受限
- 符合"batteries included"理念

---

### 决策 7: 追加式消息历史

**[FACT]** 选择：只追加，不修改历史消息

**收益**:
- LLM KV cache 友好
- 实现简单
- 完整审计轨迹

**代价**:
- 无法修正错误
- 历史可能包含冗余
- 文件持续增长

**替代方案**:
- 允许编辑历史
- 定期压缩
- 滑动窗口

**为何选择当前方案**:
- 性能优化（cache）
- 简单性
- 整合机制补偿

---

### 决策 8: MCP 集成

**[FACT]** 选择：支持 MCP 协议

**收益**:
- 标准化工具接口
- 社区生态
- 跨语言支持

**代价**:
- 额外复杂性
- 协议开销
- 调试困难

**替代方案**:
- 只支持内置工具
- 自定义插件系统
- Python 函数注册

**为何选择当前方案**:
- 未来标准
- 生态系统
- 扩展性

---

## 设计权衡总结

| 维度 | 选择 | 优势 | 劣势 |
|-----|------|------|------|
| 架构 | 单进程 | 简单 | 不可扩展 |
| 存储 | 文件系统 | 零依赖 | 性能差 |
| 并发 | 全局锁 | 正确性 | 吞吐量低 |
| Provider | LiteLLM | 多样性 | 抽象开销 |
| Memory | LLM 总结 | 智能 | 成本高 |
| Channel | 独立实现 | 灵活 | 维护重 |

## 体现 AgentOS 独特性的设计

**[INFERENCE]** 以下设计体现了 nanobot 作为 AgentOS 的特点：

1. **Multi-channel 优先** - 不是单一界面，而是多平台接入
2. **Session 隔离** - 每个聊天上下文独立状态
3. **Tool-centric** - 工具调用是核心能力
4. **Memory 系统** - 长期记忆而非单次对话
5. **Proactive 行为** - Heartbeat 和 Cron 主动执行
6. **Workspace 概念** - 文件系统作为工作空间

## 技术债与复杂度来源

**[INFERENCE]** 主要技术债：

1. **Channel 代码重复** - 11 个实现有相似逻辑
2. **全局锁** - 限制并发性能
3. **粗糙的 token 估算** - 可能不准确
4. **无结构化日志** - 难以分析
5. **缺少监控** - 无 metrics/tracing
6. **配置验证不足** - 运行时才发现错误
